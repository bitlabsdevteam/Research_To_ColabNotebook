# Sprint v1 — Walkthrough

## Summary

Sprint v1 delivers a fully functional web application called **Paper2Notebook** that converts research paper PDFs into interactive Google Colab tutorial notebooks. A user enters their OpenAI API key, uploads a PDF, and receives a downloadable `.ipynb` file with tutorial-style markdown and code cells generated by GPT-5.4. Nine of ten planned tasks were completed (all P0s); the P1 figure extraction task was deferred to v2.

### `apps/web/app/page.tsx`
**Purpose**: Main application page — orchestrates all components and the API call.

**Key logic**:
```tsx
async function handleGenerate() {
  setIsLoading(true);
  setError(null);
  setNotebook(null);

  const formData = new FormData();
  formData.append("pdf", pdfFile);
  formData.append("apiKey", apiKey);

  const res = await fetch(`${API_URL}/generate`, {
    method: "POST",
    body: formData,
  });

  if (!res.ok) {
    const body = await res.json().catch(() => null);
    throw new Error(body?.message || `Generation failed`);
  }

  const data = await res.json();
  setNotebook(data);
}
```

The page manages four pieces of state: `pdfFile`, `isLoading`, `notebook` (the result), and `error`. The `canGenerate` flag gates the button: all three conditions (key set, file selected, not loading) must be true. On success, the `ResultPanel` appears with download/Colab buttons. On failure, a red error message is shown. During loading, an animated spinner is displayed.

### `apps/api/src/generate/generate.controller.ts`
**Purpose**: HTTP endpoint `POST /generate` — accepts multipart form data (PDF file + API key), validates inputs, delegates to `GenerateService`.

**Key behavior**:
- Uses Multer's `FileInterceptor` to parse the `pdf` field from multipart form data
- Validates: file must be present, must be PDF (mimetype or extension), must be under 20 MB
- Validates: `apiKey` must be a non-empty string
- Returns 400 with descriptive message on any validation failure
- Returns 200 with the complete `.ipynb` JSON on success

The file is held in memory (Multer's default `memoryStorage`) — no temp files are written to disk.

---

### `apps/api/src/generate/generate.service.ts`
**Purpose**: Pipeline orchestrator — the single place where all backend services are called in sequence.

```typescript
async generate(pdfBuffer: Buffer, apiKey: string) {
  // 1. Parse PDF text into sections
  const parsed = await this.pdfParser.parse(pdfBuffer);

  // 2. Generate notebook cells via GPT-5.4
  const cells = await this.aiService.generateNotebook(
    parsed.sections, [], apiKey
  );

  // 3. Build .ipynb notebook
  const notebook = this.notebookBuilder.build(cells);

  return notebook;
}
```
The pipeline is intentionally linear: parse → generate → build. Figures are passed as an empty array `[]` because Task 6 (figure extraction) was deferred. The service receives all three dependencies via NestJS constructor injection.

### `apps/api/src/pdf-parser/pdf-parser.service.ts`
**Purpose**: Extracts text from PDF buffers and identifies academic paper sections by heading patterns.

**Key functions**:
- `parse(buffer)` — Uses `pdfjs-dist` to extract text page-by-page, then calls `extractSections()`
- `extractSections(text)` — Splits text into lines, matches each line against heading patterns, groups content between headings into sections

The section detection uses three regex patterns:
```typescript
const SECTION_PATTERNS = [
  /^(Abstract)\s*$/im,                                    // "Abstract" standalone
  /^(\d+\.?\d*\.?\s+[A-Z][A-Za-z\s&-]+)\s*$/m,         // "1. Introduction"
  /^(Introduction|Background|Methods|Results|...)\s*$/im, // Common unnumbered headings
];
```
Text from `pdfjs-dist` comes as space-separated strings from each text item on the page. The service splits on double-spaces and newlines to reconstruct line boundaries, then scans for headings. If no sections are found, the entire text is returned as a single "Full Text" section.

### `apps/api/src/ai/ai.service.ts`
**Purpose**: Sends structured prompts to GPT-5.4 and parses the response into notebook cells.

**Key logic**:
```typescript
const client = new OpenAI({ apiKey });
const response = await client.chat.completions.create({
  model: "gpt-5.4",
  messages: [
    { role: "system", content: SYSTEM_PROMPT },
    { role: "user", content: userPrompt },
  ],
  temperature: 0.3,
});
const cells = JSON.parse(response.choices[0].message.content);
```

A new `OpenAI` client is created per-request using the user's API key — the key is never stored server-side. Temperature is set to 0.3 for more deterministic, structured output. The service expects GPT to return a JSON array of `{ cell_type, source }` objects and throws descriptive errors for API failures, empty responses, or malformed JSON.

### `apps/api/src/ai/prompts/notebook-prompt.ts`
**Purpose**: Contains the system prompt and user prompt builder for the GPT-5.4 notebook generation call.

The **system prompt** instructs GPT to:
1. Start with a title and overview markdown cell
2. Create paired markdown/code cells for each section
3. Include `pip install` cells at the top
4. Add inline code comments
5. Output ONLY valid JSON (no surrounding text)

The **user prompt builder** (`buildUserPrompt`) formats the extracted paper sections as markdown headings with content, appends figure metadata if available, and ends with the generation instruction.

## Data Flow

1. **User enters API key** → Stored in `ApiKeyContext` (React state, browser memory only)
2. **User uploads PDF** → `PdfUpload` validates (`.pdf` extension, <20 MB) → stored in `pdfFile` state
3. **User clicks "Generate Notebook"** → `handleGenerate()` creates a `FormData` with the PDF blob and API key string
4. **Frontend sends `POST /generate`** → Multipart form data over HTTP to NestJS on port 3001
5. **`GenerateController`** → Multer parses the file, validates PDF mimetype and apiKey presence
6. **`GenerateService.generate()`** orchestrates the pipeline:
   - **Step 1**: `PdfParserService.parse(buffer)` → Uses `pdfjs-dist` to extract text page by page → Regex-matches section headings → Returns `{ rawText, pageCount, sections[] }`
   - **Step 2**: `AiService.generateNotebook(sections, [], apiKey)` → Builds system + user prompt → Sends to GPT-5.4 → Parses JSON response → Returns `NotebookCell[]`
   - **Step 3**: `NotebookBuilderService.build(cells)` → Converts cells to nbformat 4.5 → Adds Colab metadata → Returns complete `.ipynb` JSON object
7. **Controller returns the notebook JSON** as the HTTP response body (200)
8. **Frontend receives JSON** → Sets `notebook` state → `ResultPanel` appears
9. **User clicks "Download .ipynb"** → `downloadNotebook()` creates a Blob, triggers browser download
10. **User clicks "Open in Colab"** → Downloads the file + opens `colab.research.google.com/#create=true` in a new tab

## Security Measures

- **API key never persisted**: Stored only in React state (browser memory), sent only with generation requests, never written to localStorage/cookies/disk
- **API key never logged server-side**: The key is passed through to OpenAI and not stored or logged
- **File validation**: PDF mimetype and extension checked server-side (Multer filter), file size limited to 20 MB
- **Input validation**: API key presence validated server-side before processing
- **CORS enabled**: Backend allows cross-origin requests for frontend communication
- **No file persistence**: Uploaded PDFs are held in memory only (Multer `memoryStorage`), never written to disk
- **Dependency audit**: `npm audit` run after every task — only pre-existing dev-only vulnerabilities in `@nestjs/cli` transitive deps (ajv ReDoS in Angular devkit schematics, not shipped to production)

## Known Limitations

1. **No figure extraction** (Task 6 deferred): The pipeline passes an empty figures array to the AI service. Papers with important diagrams/charts won't have those visuals in the notebook.
2. **No streaming**: The frontend blocks with a spinner while GPT generates the entire notebook. For long papers, this could take 30-60+ seconds with no progress indication.
3. **"Open in Colab" is two-step**: The user must manually upload the downloaded file to Colab because local blobs can't be passed directly. A GitHub Gist integration would enable one-click opening.
4. **Section detection is regex-based**: The heading detection works well for standard academic papers but may miss non-standard heading formats, multi-line headings, or papers without numbered sections.
5. **PDF text extraction quality varies**: `pdfjs-dist` extracts text by rendering items, which can lose formatting nuance (columns, tables, equations). Mathematical notation may not extract cleanly.
6. **No error recovery**: If the OpenAI API call fails, the user must click "Generate" again. There's no retry logic or partial result caching.
7. **Single model hardcoded**: The `gpt-5.4` model is hardcoded. Users can't choose a cheaper/faster model.
8. **No rate limiting**: The backend doesn't rate-limit requests, which could lead to abuse if deployed publicly.
9. **API key sent in form body**: The key travels in the POST body (not a header), which is fine for HTTPS but less conventional for API key transmission.

## What's Next

Recommended priorities for Sprint v2:

1. **P0 — Figure extraction** (deferred Task 6): Extract embedded images from PDFs using `pdfjs-dist`'s `getOperatorList()` or a library like `pdf2pic`, and include them in the notebook
2. **P0 — Streaming generation**: Use Server-Sent Events (SSE) to stream notebook cells as they're generated, showing progressive output instead of a blocking spinner
3. **P1 — GitHub Gist integration**: Auto-create a Gist with the notebook and construct a `colab.research.google.com/gist/...` URL for true one-click Colab opening
4. **P1 — Model selector**: Let users choose between different OpenAI models (GPT-5.4 vs cheaper/faster options) from the frontend
5. **P1 — Better PDF parsing**: Add support for two-column layouts, LaTeX equation extraction, and table parsing
6. **P2 — Rate limiting and abuse prevention**: Add request throttling to the backend
7. **P2 — Paper history**: Optional localStorage-based history of previously generated notebooks
